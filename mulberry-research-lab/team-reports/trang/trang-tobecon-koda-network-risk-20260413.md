## 리스크 정의

**상황**: Koda CTO가 상위 모델(고성능 AI)을 사용하더라도,
네트워크 연결이 불안정하면 개발 파이프라인 전체가 지연될 수 있다.

**취약 구간 (단 하나)**:

```
[Koda 로컬] ──네트워크──▶ [GitHub push]  ← 여기만 취약
                                ↓
                         [Colab] ──────▶ [Railway]
                         (Google Cloud)  (클라우드)
```

Colab·Railway는 클라우드 기반이라 Koda 네트워크와 무관하게 안정적.
**Koda 로컬 → GitHub push 구간만이 유일한 취약 포인트다.**

---

## Railway Agent 역량 확인 (긍정 요소)

Railway Agent는 자기 영역에서 신뢰할 수 있다:

| 기능 | 상태 |
|------|------|
| 자동 배포 (GitHub → Railway) | ✅ 안정적 |
| **Socket.IO 지원** | ✅ 확인됨 |
| 환경 변수 자동 설정 | ✅ |
| 실시간 모니터링·로그 | ✅ |
| 배포 실패 자동 감지 | ✅ |
| 네트워크 독립 운영 | ✅ (Koda 네트워크와 무관) |

→ Railway는 코드가 GitHub에 올라오는 순간부터 완전 자율 운영 가능.

## 네트워크 불안정 대비 3대 전략

### 전략 1 — 짧게·자주 Push (소단위 커밋 원칙)

```
기존 방식:
   기능 완성 (긴 작업) → 한 번에 대형 push
   → 네트워크 끊기면 전부 로컬에 갇힘

개선 방식:
   함수 단위 / 파일 단위로 자주 push
   → 네트워크 끊겨도 마지막 push까지는 GitHub에 보존
   → Colab·Railway가 그 지점부터 이어서 진행 가능
```

**규칙**: 1~2시간 작업 후 반드시 push. 미완성이어도 WIP 커밋 허용.

---

### 전략 2 — Colab Agent를 보조 개발자로 운영

```
Koda 네트워크 정상 시:
  Koda (핵심 설계·구현) → GitHub push → Colab (리팩토링)

Koda 네트워크 불안정 시:
  마지막 push 기준으로 Colab Agent 독립 진행
  → 리팩토링·테스트·문서화 계속 수행
  → Koda 복귀 후 PR 병합만 하면 됨
```

**Colab Agent 독립 수행 가능 작업**:
- 코드 리팩토링 (기존 코드 기반)
- 유닛 테스트 작성
- API 명세 문서화
- 네이버 SMTP / Notion API 연동 모듈 작성
- 카카오 알림톡 템플릿 코드 초안

---

### 전략 3 — GitHub Copilot 로컬 캐시 활용

```
Koda 환경:
  GitHub Copilot → 네트워크 간헐적이어도 로컬 캐시로 제안 가능
  → 코드 작성 자체는 계속 가능
  → 네트워크 복구 순간 즉시 push
```

**핵심**: 네트워크가 없어도 코딩은 계속, push만 기다리는 구조.

## 비상 시나리오별 대응

| 시나리오 | 영향 | 대응 |
|---------|------|------|
| Koda 네트워크 1~2시간 단절 | push 지연 | Colab Agent 독립 작업 계속 |
| Koda 네트워크 반일 이상 단절 | 스프린트 지연 | 대표님 보고 → 일정 조정 |
| GitHub 서비스 장애 | push 불가 | 로컬 저장 후 복구 시 일괄 push |
| Railway 장애 | 배포 중단 | Railway 상태 페이지 확인 → 자동 복구 대기 |
| Colab 세션 만료 | 작업 중단 | 세션 재시작 (코드는 GitHub에 보존) |

